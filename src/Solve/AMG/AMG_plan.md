# AMG Review and Implementation Plan

## 1. Critical bugs

### 1.1 Fine-level matrix never refreshed from current A on CUDA — hierarchy freezes on first solve
`7_AMG_update.jl:10-19`, `5_AMG_cycle.jl:39`, `ext/XCALibre_CUDAExt.jl:48-58`

On CUDA, `_amg_setup_matrix(A_gpu, CPU())` copies the GPU sparse matrix into a CPU `SparseXCSR` at first build. `level[1].A` is that CPU copy, not a view into the GPU matrix. On subsequent solves `update!` calls `refresh_hierarchy!`, which only refreshes `level.diagonal`, `level.inv_diagonal`, `level.lambda_max` from `level.A` — it never re-syncs `level.A`'s `nzval` from the current GPU `A`. Consequence: for transient PISO the pressure matrix changes every step; the AMG hierarchy permanently uses step-1 values on CUDA. Almost certainly a major accuracy driver in the CUDA smoke runs.

Fix: in `update!`, when the pattern matches, copy the current `_nzval(A)` back into the CPU-side `level[1].A.nzval` before refreshing diag.

### 1.2 Coarse-level operators are never regenerated during reuse (frozen Galerkin)
`7_AMG_update.jl:10-20`, `1_AMG_setup.jl:109-116`

`refresh_hierarchy!` only updates the finest level's diag/invdiag/λmax. `levels[k].A` for `k ≥ 2`, and all `P`/`R` remain pinned to whatever was built on the first pattern match. Even on CPU (where `level[1].A` tracks the user matrix by reference), the coarse Galerkin operators `A_{k+1} = R_k A_k P_k` are frozen to the first build's values. For PISO pressure (where the operator changes every step through `rDf` and mass-flux coefficients) this degrades the hierarchy steadily. Likely a primary reason direct AMG lags `Cg()+Jacobi()` on the cylinder case.

Fix options (in order of effort):
1. Re-Galerkin the coarse operators on every `refresh_hierarchy!` using the cached P/R.
2. Full re-setup (rebuild P too) every N solves or whenever a convergence sensor trips.
3. Lazy adaptive re-setup driven by V-cycle convergence rate (classic adaptive AMG heuristic).

### 1.3 Cached coarse-solver factorisation goes stale
`1_AMG_setup.jl:153`, `5_AMG_cycle.jl:1-6`

`last_level.coarse_solver = factorize(_csr_to_csc(last_level.A))` is set at setup. `refresh_hierarchy!` never refactorises; on reuse, `_coarse_solve!` uses the stale factorisation. Coupled with 1.2 this is self-consistent (stale coarse op → stale factor), but once we fix 1.2 we must refactorise at the same time.

### 1.4 AMGChebyshev conflates polynomial degree with pre/post-sweep count
`4_AMG_smoothers.jl:31-50`

The loop is `for _ in 1:max(loops, smoother.degree)`. With `presweeps=postsweeps=10` (the current default via `smoothing_steps=10`) and `degree=2`, the Chebyshev recurrence runs **10 contiguous steps** without resetting α/β. Semantically this is neither "10 Chebyshev-degree-2 applications" nor "a single degree-10 Chebyshev polynomial" — the coefficients are generated by the degree-2 three-term recurrence but extended to step 10, producing an undefined polynomial that depends on truncation.

Fix: `degree` is the polynomial degree used inside one smoothing pass; `loops` is the number of passes. Correct structure:
```
for _ in 1:loops
    # reset α = 1/d, β = 0, p = 0
    for k in 1:degree
        # Chebyshev three-term recurrence
    end
end
```
Production AMG treats Chebyshev smoothing as a single pass at a fixed low degree (e.g. 2–4), and the degree is the only controlling parameter. The current `smoothing_steps=10` for Chebyshev is almost certainly over-smoothing.

### 1.5 `amg_cg_solve!` runs `_is_symmetric(A)` on every call
`6_AMG_cg.jl:1-14, 19`

`_is_symmetric` is O(nnz × avg_row). On a 64k-row pressure matrix that's a full sparsity walk on every pressure solve — dominates many CG mode runs. It is also invoked inside the CUDA fallback path (the CPU copy is re-checked).

Fix: cache symmetry status on the hierarchy (compute once during setup), or add a `assume_symmetric::Bool` flag to `AMG(...)` (default true for pressure use cases).

### 1.6 `_amg_cpu_solve!` reallocates the CPU workspace on every call
`5_AMG_cycle.jl:38-63`

On CUDA, every pressure solve builds a fresh `AMGWorkspace` with 5 `similar` buffers plus two `Array(b)`/`Array(x)` copies. Aggregate cost dominates on small/medium problems.

Fix: attach a CPU-mirror workspace to the hierarchy object at setup time and reuse it.

### 1.7 AMG-dispatch `solve_system!` drops features of the generic path
`7_AMG_update.jl:23-38` vs `Solve/Solve_1_api.jl:268-307`

The generic `solve_system!` does:
- `apply_smoother!(setup.smoother, values, A, b, hardware)` — pre-equation Jacobi smoothing.
- Crank–Nicolson explicit reconstruction (`2*x - values`).
- `@warn` on hitting `itmax`.

The AMG dispatch skips all three. Pressure happens not to use Time{CrankNicolson}, so CN isn't currently triggered — but the moment someone points AMG at a field with CN time, the update is silently wrong. Silent `itmax` hit is also a subtle convergence bug.

Fix: align the AMG `solve_system!` with the generic path (apply_smoother call, CN step, itmax warning). The `preconditioner=Jacobi()` in the user API is currently *required* by `SolverSetup` but ignored by AMG — document this, or allow `preconditioner=nothing` when `solver isa AMG`.

### 1.8 Pattern check is weak
`7_AMG_update.jl:11`

`hierarchy.nrows != _m(A) || hierarchy.nnz != length(_nzval(A))` misses cases where rows and nnz are unchanged but `colval`/`rowptr` differ (mesh reordering, BC topology change). Production AMG hashes the sparsity pattern or checks `rowptr`/`colval` pointer equality.

## 2. Correctness / algorithmic limitations

These are not bugs in the strict sense, but the current algorithms are significantly weaker than the names suggest.

### 2.1 `RugeStuben` C/F splitting is missing Pass 2
`2_AMG_coarsening.jl:74-87`

`_coarse_fine_split` runs a single greedy pass: highest-weight undecided node becomes C, its strong neighbours become F. Classical Ruge–Stüben requires a **second pass** to enforce that every strong F-F connection has a common C neighbour (otherwise direct interpolation is ill-posed). Without Pass 2 the interpolation operator represents smooth error poorly.

### 2.2 `RugeStuben` interpolation uses uniform weights
`2_AMG_coarsening.jl:120-125`

`weight = inv(length(neigh))` — pure injection-averaged interpolation. Classical RS direct interpolation uses operator-dependent weights:

    w_ij = -(a_ij + Σ_{m∈F_i^s} a_im · a_mj / Σ_{k∈C_i} a_mk) / a_ii

The current form discards operator information entirely and is much closer to constant extension than to RS.

### 2.3 `SmoothAggregation` aggregation is naive pairwise-first-neighbour
`2_AMG_coarsening.jl:26-40`

`_pairwise_aggregates` claims node i and its first unaggregated strong neighbour as a pair. Production SA (PyAMG `StandardAggregation`, hypre, ML/Trilinos) uses a three-phase algorithm: (1) seed aggregates by MIS-like traversal, (2) enlarge by strong neighbours, (3) handle leftovers. The current form creates many very small aggregates (including singletons) and gives irregular coarsening rates.

### 2.4 No near-nullspace candidates
`2_AMG_coarsening.jl:63-72`, `1_AMG_setup.jl:138-140`

Tentative P is a piecewise-constant indicator from `agg`. For scalar diffusion with Dirichlet BCs the constant candidate is actually near-nullspace, so this is fine for pressure Poisson. But the framework has no way for the user to supply candidates (rigid-body modes for elasticity, multiple constant vectors for systems, etc.) and no candidate-improvement step (αAMG / adaptive SA).

### 2.5 Smoothers are limited to weighted Jacobi and Chebyshev
`4_AMG_smoothers.jl`

No (symmetric) Gauss–Seidel, no SOR, no `l1`-Jacobi, no block/overlapping smoother. SGS is the canonical AMG smoother for scalar diffusion and is typically the strongest on CPU. `l1`-Jacobi is the typical choice for GPU-resident smoothing.

### 2.6 Single cycle type
`5_AMG_cycle.jl`

Only V-cycle exists. W-cycle and K-cycle (AMG-preconditioned FCG inner) are standard tools for difficult operators and are cheap to add once the level structure is stable.

### 2.7 Coarse-grid direct solver is always LU on CSC
`1_AMG_setup.jl:153`, `5_AMG_cycle.jl:1-6`

LU (SuiteSparse) is fine for `max_coarse_rows ≈ 100–500`. For larger coarse problems, direct factorisation dominates. No fallback to iterative coarse solve, no redundant Cholesky where SPD.

### 2.8 Scalar-only framework
No concept of variable blocks / systems AMG. Not needed for pressure today, but it blocks the road to using AMG on momentum.

## 3. Performance bottlenecks

Ranked by expected cost on the 64k cylinder case.

1. **Fine SpMV and smoother inner loops are serial CPU code.** `_apply_level_smoother!(AMGJacobi)` and `_residual!` are hand-written serial CSR walks. Threading was attempted and reverted; the revert was right, but untreaded the fine level is the hot path. Moving these to KernelAbstractions kernels (matches the rest of XCALibre) unlocks multi-threading on CPU and GPU residency.
2. **Stale coarse operators (1.2) force extra V-cycles per solve.** Every unnecessary V-cycle costs the full hierarchy SpMV chain plus the coarse LU solve. Fixing 1.2 likely reduces the V-cycle count more than any micro-optimisation.
3. **CUDA fallback reallocations (1.6).** Hot-path allocations defeat GPU pipeline benefits.
4. **Repeated `_is_symmetric` (1.5).** O(nnz) per solve in CG mode.
5. **Setup SpGEMM via `sparse(I,J,V,...)`.** `_csr_to_csc(current_A) * P` rebuilds triplets and sparsifies on every setup. Once setup becomes more frequent (needed to fix 1.2), this becomes significant. Classical AMG uses pattern-preserving in-place Galerkin or symbolic-then-numeric SpGEMM.
6. **`strong::Vector{Vector{Int}}` in strength graph** allocates one Vector per row. Flat CSR-form strength graph is cheaper and cache-friendly.
7. **`R = transpose(P)` is a lazy Transpose wrapper.** SparseArrays has fast paths, but building R as a concrete CSR matrix (and storing it) avoids indirection and is friendlier to GPU SpMV backends.
8. **`_prolongate_add!` does two passes** (`mul!(tmp, P, coarse_x)`; `x += tmp`). `gemv!`-style `y += αPx` kernel fuses these.
9. **Factorisation of coarse operator is refreshed too aggressively once we fix 1.3.** Cache both the *symbolic* factorisation (pattern) and refactorise only numerics.

## 4. Integration with PISO / SIMPLE

Observations from `Solvers_2_PISO.jl`, `Solvers_1_SIMPLE.jl`, `Solvers_2_CPISO.jl`, `Solvers_1_CSIMPLE.jl`:

- `update!` is called at *every* pressure solve via the AMG `solve_system!` dispatch. Combined with bugs 1.1–1.3 this means each subsequent pressure solve uses a hierarchy whose fine-level matrix is stale (CUDA) and/or whose coarse operators are stale (CPU). That is the most likely root cause of the poor pressure residuals vs `Cg()+Jacobi()` on cylinder.
- Non-orthogonal correctors in PISO call `solve_system!` directly (not `solve_equation!`), so the same AMG dispatch is exercised. OK — but it means each non-ortho corrector redoes the AMG setup check. Cheap now, non-cheap once 1.2 is fixed.
- `solve_equation!` runs `update_preconditioner!` on `phiEqn.preconditioner` before `solve_system!`. For AMG this is wasted work (the Jacobi preconditioner in `SolverSetup` is allocated and updated but never consulted). Consider detecting AMG and skipping, or making `preconditioner` optional when `solver isa AMG`.
- The AMG API exposes `mode=:solver` and `mode=:cg`. In PISO/SIMPLE there is a third natural mode — AMG-preconditioned **Bicgstab** — for momentum or for non-symmetric pressure operators (rotating reference frame, strong anisotropy). Not currently available.

User-facing API (`examples/2D_cylinder_U_AMG.jl`): surface-level the AMG options are reasonable. Deeper, these user knobs mostly map to the right internal parameters, except `smoothing_steps=10` applied to Chebyshev is interpreted as 10 polynomial iterations (see 1.4). Document or enforce the mapping.

## 5. Prioritised implementation pathway

Each stage ends with a code state that is a strict improvement over the previous one, testable on the cylinder case against the `Cg()+Jacobi()` baseline.

### Stage A — Fix correctness and stop wasting work (1–2 days)

**A1.** Fix 1.1: re-sync `level[1].A.nzval` from the current (possibly GPU) `A` on every `update!`. This alone should visibly move CUDA pressure residuals.
**A2.** Fix 1.2 + 1.3: in `refresh_hierarchy!`, recompute coarse operators by re-applying cached P/R as `A_{k+1} := R_k · A_k · P_k`, and refactorise the coarse operator at the bottom. Keep the "full rebuild on pattern change" fallback.
**A3.** Fix 1.4: separate polynomial degree from sweep count; run the Chebyshev recurrence `presweeps` times, each of length `degree`. Adjust default `smoothing_steps` for Chebyshev (typically 1 pass of degree 2 or 3).
**A4.** Fix 1.5 + 1.6: cache symmetry flag on hierarchy; cache CPU-mirror workspace on hierarchy.
**A5.** Fix 1.7: align AMG `solve_system!` with the generic path (apply_smoother, CN, itmax warn).
**A6.** Fix 1.8: strengthen the pattern check to `rowptr`/`colval` equality (cheap: pointer check or hash-on-setup).

Exit criterion: direct-AMG pressure residuals on cylinder are within (say) 1 order of magnitude of `Cg()+Jacobi()` in matched short runs; CUDA smoke run produces residuals consistent with CPU.

### Stage B — Production-grade classical algorithms (1 week)

**B1.** Proper classical Ruge–Stüben:
  - Pass-2 C/F splitting (enforce strong F-F connectivity via a C point).
  - Operator-dependent direct interpolation (use `a_ij` weights).
  - Reference: Ruge & Stüben (1987); PyAMG `ruge_stuben_solver` internals.

**B2.** Proper Smoothed Aggregation:
  - Replace `_pairwise_aggregates` with `StandardAggregation` (MIS-seeded three-phase).
  - Tentative P from user-supplied near-nullspace candidates (default: constant vector).
  - Jacobi-smoothed P (already present, keep).
  - Reference: Vaněk, Mandel & Brezina (1996); PyAMG `smoothed_aggregation_solver`.

**B3.** Symmetric Gauss–Seidel and l1-Jacobi smoothers. SGS for CPU default; l1-Jacobi prepared for GPU. Chebyshev remains for anisotropic or GPU-heavy cases.

**B4.** W-cycle option (parameter on `AMG(...)`). Single-liner given the existing recursion.

**B5.** Adaptive re-setup trigger: if V-cycle convergence rate exceeds a threshold, force a full rebuild. Classical defensive pattern against stale hierarchies.

Exit criterion: direct AMG and AMG-CG beat `Cg()+Jacobi()` on cylinder at equal wall-clock; hierarchy quality diagnostics (operator complexity, grid complexity, convergence factor) are logged.

### Stage C — Performance and GPU residency (1–2 weeks)

**C1.** Port fine-level SpMV, smoothers, residual, restriction, prolongation to KernelAbstractions `@kernel` with CPU-threaded and CUDA/AMD/oneAPI backends (matches existing XCALibre conventions). This is the key step for genuine GPU AMG.

**C2.** Setup on GPU:
  - GPU strength graph (per-row reductions).
  - GPU aggregation (parallel MIS — e.g. Luby-style or Bell–Dalton).
  - GPU SpGEMM for Galerkin (use CUSPARSE for CUDA, generic for others).
  - Build P/R as concrete CSR on-device.

**C3.** Preallocate coarse-level sparsity: on first setup, record symbolic patterns of all `A_k`, P/R, and preallocate. On re-setup with pattern match, only values are refilled — no triplet buffers, no `sparse(I,J,V,...)`.

**C4.** Concrete `R` storage (not lazy `transpose(P)`). Makes GPU SpMV path uniform with CPU.

**C5.** Coarse solver choices: LU (current), sparse Cholesky (SPD fast path), and a bounded-iteration Chebyshev fallback for when `max_coarse_rows` is large.

Exit criterion: GPU AMG end-to-end (no CPU fallback); setup cost scales ≤ O(nnz) and is reusable; AMG setup stops showing up in flame graphs of long runs.

### Stage D — Modern / novel techniques (research-grade, as needed)

**D1.** K-cycle (AMG-preconditioned FCG inside each level) — robust against difficult operators with minimal additional code.

**D2.** Energy-minimisation interpolation (EMIN, Brezina et al.). Replaces smoothed-aggregation P-smoothing with a short CG minimisation. Markedly better for anisotropic diffusion.

**D3.** Adaptive AMG / αAMG: iteratively improve near-nullspace candidates. Useful when the user cannot supply good candidates a priori.

**D4.** Compatible relaxation (Brandt) as a coarsening quality sensor, used to drive re-setup decisions in the adaptive trigger of B5.

**D5.** Block / systems AMG to enable AMG on momentum. Block-SA with per-variable aggregation, or unknown-based separate hierarchies.

**D6.** Multi-colour (ω-)Gauss–Seidel for GPU — required if users want GS-class smoothing on device.

**D7.** Low / mixed precision V-cycles (FP32 smoothers with FP64 residual correction). State-of-the-art for GPU AMG (hypre, AmgX).

## 6. Pragmatic alternative: wrap AlgebraicMultigrid.jl

Worth naming explicitly: `AlgebraicMultigrid.jl` already provides well-tested classical Ruge–Stüben and smoothed-aggregation solvers, Jacobi / GS / Chebyshev smoothers, and integrates with `KrylovPreconditioners.jl`. Wrapping it as a thin backend behind the existing `AMG`/`SmoothAggregation`/`RugeStuben` API would:

- immediately close most of the gaps in Stage B,
- retain XCALibre's user-facing configuration surface,
- not solve GPU residency (it is CPU only),
- reduce long-term maintenance of the coarsening/interpolation code.

Recommended as a **parallel track**, not a replacement for the staged plan: wrap it as an `AMG(backend=:amgjl)` option for users who need robust CPU AMG today, while the native path evolves towards GPU residency (Stage C onward).

## 7. Suggested order of attack

1. Stage A in one PR — all bugfixes, nothing new algorithmically.
2. Decide between "grow native" (Stage B) and "wrap AlgebraicMultigrid.jl" for the CPU path.
3. Stage C regardless of the above (GPU residency is not in AlgebraicMultigrid.jl).
4. Stage D items only as concrete user needs emerge.

This keeps every intermediate state runnable, keeps the cylinder case as a live regression target, and defers novel techniques until the classical foundation is solid.
